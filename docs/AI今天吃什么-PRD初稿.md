## 1. 产品概述

### 1.1 项目背景

现代用户在日常用餐场景中经常遇到“选择困难”：外卖平台信息过载、随机决策工具缺少真实场景、优惠入口与推荐结果割裂。`AI今天吃什么` 旨在通过微信小程序提供一个快速、有趣、可落地的用餐决策入口，结合 AI 推荐、LBS 附近商家、盲盒玩法、自定义菜单与外卖红包导流，帮助用户更快决定吃什么。

### 1.2 产品定位

`AI今天吃什么` 是一款面向日常用餐决策的微信小程序，核心定位为：

* 快速决策：一键获得今日推荐或转盘/翻卡片结果。
* 智能匹配：结合位置、偏好、预算、历史反馈推荐商家或菜品。
* 实惠省钱：推荐链路中整合美团、饿了么等红包入口。
* 趣味惊喜：通过转盘、翻卡片、盲盒卡片增强使用粘性。

### 1.3 产品名称

* 小程序名称：`AI今天吃什么`
* 小程序简称：`AI吃什么`
* 项目代号：`WhatToEat-AI`

## 3. 目标用户

### 3.1 核心用户

* 18-35 岁城市上班族、学生。
* 每天在午餐、晚餐、夜宵场景中需要快速决定吃什么。
* 经常使用外卖平台，对优惠券、红包敏感。

### 3.2 次要用户

* 家庭烹饪者，需要基于已有食材决定做什么菜。
* 健身、减脂用户，需要按饮食目标筛选菜品。
* 喜欢转盘、抽卡、盲盒等轻游戏化决策方式的用户。

### 3.3 典型用户场景

* 早餐通勤：用户上班路上想快速决定吃点什么，希望 1 个明确推荐，决策时间不超过 5 秒。
* 工作日午餐：用户不想浏览大量商家，希望快速获得 1-3 个推荐。
* 下午茶/饮品：用户想喝点什么或买点甜品/烘焙，希望快速决定品类与门店。
* 夜宵：用户想通过转盘或卡片获得带惊喜感的夜宵选择。
* 家里做饭：用户输入已有食材，系统推荐可做菜品。
* 外卖优惠：用户想先领红包，再去平台下单。

## 4. 产品目标

### 4.1 MVP 目标

第一版小程序应完成“能用、好玩、可导流”的核心闭环：

* 用户可以在首页通过 AI/规则推荐获得吃什么建议。
* 用户可以查看附近餐饮或外卖平台入口。
* 用户可以使用转盘或翻卡片玩法进行随机决策。
* 用户可以管理自定义菜单。
* 用户可以通过红包入口跳转至外卖平台。

### 4.2 成功指标

* 首页推荐点击率。
* 推荐结果确认率。
* 红包入口点击率。
* 翻卡片/转盘玩法使用次数。
* 自定义菜单创建率。
* 次日留存和 7 日留存。

## 6. 功能需求

### FR-01 用户登录与身份

**目标**：建立用户唯一身份，用于偏好、历史记录和反馈。

**需求**：

* 支持微信登录。
* 获取用户 openId，后端生成用户记录。
* 用户可授权头像、昵称。
* 记录首次访问时间、最近访问时间。

**验收标准**：

* 新用户首次进入后可生成唯一用户标识。
* 未授权头像昵称时，小程序仍可使用基础推荐和随机玩法。

### FR-02 首页推荐

**目标**：帮助用户快速获得“今天吃什么”的结果。

**需求**：

* 首页展示双模式 Tab：`听天由命`、`附近餐饮`。
* `听天由命` 模式展示转盘或一键推荐结果。
* `附近餐饮` 模式展示查看附近美食入口和推荐商家卡片。
* 推荐结果包含名称、推荐理由、标签。
* 推荐结果支持点赞/点踩反馈。

**推荐逻辑**：

* MVP 使用规则引擎：
  * 距离权重
  * 评分权重
  * 预算匹配
  * 用户口味偏好
  * 历史反馈过滤
* 二期接入协同过滤 + GBDT 排序模型。

**验收标准**：

* 用户点击推荐按钮后，2 秒内展示结果。
* 推荐结果至少包含 1 条，最多 3 条。
* 第三方接口失败时展示本地内置菜品推荐。

### FR-03 定位与附近餐饮

**目标**：基于用户位置获取附近餐饮 POI。

**需求**：

* 用户授权地理位置后，请求附近 1-3 公里餐饮商家。
* 支持高德或百度地图 API。
* 商家信息包含名称、地址、距离、经纬度、评分、人均价格、图片。
* 商家数据缓存 1 小时。

**验收标准**：

* 用户拒绝定位时，提示可使用“听天由命”和自定义菜单。
* 地图 API 不可用时，使用缓存或默认推荐。

### FR-04 商家详情与外卖跳转

**目标**：将推荐结果转化为真实下单行为。

**需求**：

* 商家卡片可进入详情页。
* 详情页展示地址、电话、营业时间、图片、人均价格、评分。
* 提供“去美团下单”“去饿了么下单”等按钮。
* 跳转链接带 CPS 参数。

**验收标准**：

* 点击跳转按钮能唤起对应小程序或 H5。
* 后台记录跳转事件。

### FR-05 红包与 CPS 导流

**目标**：把优惠入口自然嵌入用餐决策链路。

**需求**：

* 首页和吃喝玩乐页展示红包卡片轮播。
* 红包卡片包含平台 Logo、标题、描述、按钮。
* 支持复制口令、立即领取、跳转小程序/H5。
* 后台可配置红包标题、图片、链接、排序、上下线状态。

**验收标准**：

* 红包卡片点击可记录事件。
* 红包活动失效时不展示或自动降级。

### FR-06 吃喝玩乐页

**目标**：承载平台活动、品牌系列和附近优惠玩法。

**需求**：

* 顶部展示红包活动轮播。
* 展示“大牌系列转盘”卡片列表。
* 卡片包含图片、标题、标签。
* 点击卡片进入对应主题转盘页。
* 提供“附近吃喝玩乐”入口。

**验收标准**：

* 卡片可配置化。
* 页面支持两列卡片布局，适配主流手机屏幕。

### FR-07 翻卡片/盲盒玩法

**目标**：提供轻游戏化随机决策体验。

**需求**：

* 翻卡片页展示 3x3 卡片矩阵。
* 显示剩余翻卡次数。
* 点击卡片后翻开并展示结果。
* 支持灰色空卡位。
* 支持“分享”和“再来一局”。
* 支持创建盲盒卡片，选择菜单来源。

**限制规则**：

* 每轮最多翻卡次数可配置，默认 3 次。
* 可通过分享或激励视频获得额外次数。
* 避免诱导分享，分享重置逻辑需符合微信审核规范。

**验收标准**：

* 翻卡后结果不重复。
* 剩余次数正确递减。
* 次数耗尽后禁用未翻开卡片。

### FR-08 转盘玩法

**目标**：通过视觉化转盘降低决策成本。

**需求**：

* 首页支持“听天由命”转盘。
* 主题页支持分类转盘，例如夜宵、饮品、轻食。
* 支持“震动反馈”开关。
* 支持开始转盘、停止并展示结果。
* 转盘选项来自内置菜单或用户自定义菜单。

**验收标准**：

* 转盘动画流畅。
* 停止后明确展示选中项。
* 用户可以切换菜单分类。

### FR-09 自定义菜单管理

**目标**：允许用户创建自己的菜品池，提高个性化。

**需求**：

* 菜单列表展示用户全部菜单。
* 支持添加新菜单。
* 支持编辑菜品、删除菜单、分享给好友。
* 菜单包含名称、菜品列表、分类、是否公开。
* 内置默认菜单：早餐菜单、午餐菜单、下午茶/饮品菜单、晚餐菜单、夜宵菜单、丰富主义者。
* "减脂菜谱"从内置菜单移至个人中心的偏好筛选项（详见 FR-11），不在场景/菜单维度并列。

**验收标准**：

* 用户可创建至少 20 个菜单。
* 单个菜单可包含至少 100 个菜品。
* 删除前需要二次确认。

### FR-10 食材选择推荐

**目标**：帮助用户根据已有食材决定做什么。

**需求**：

* 食材选择页展示分类与食材按钮。
* 支持必要食材和可选食材两种选择状态。
* 食材分类包括肉禽类、水产类、蔬菜类、豆类/豆制品/面食等。
* 根据选择匹配内置菜品库。
* 结果展示菜名、所需食材、简单做法。

**验收标准**：

* 必要食材必须包含在匹配结果中。
* 无匹配结果时给出替代建议。

### FR-11 个人中心与偏好

**目标**：沉淀用户个性化信息和行为历史。

**需求**：

* 展示用户头像、昵称。
* 支持设置口味偏好、辣度、忌口、预算范围。
* 展示历史推荐记录。
* 支持对历史结果重新点赞/点踩。
* 提供反馈入口。

**验收标准**：

* 偏好保存后影响后续推荐排序。
* 历史记录可按时间倒序查看。

### FR-12 分享功能

**目标**：提升传播与玩法复用。

**需求**：

* 支持分享推荐结果。
* 支持分享翻卡片结果。
* 支持分享自定义菜单。
* 支持生成分享卡片图。

**验收标准**：

* 分享文案、图片符合微信审核规范。
* 分享不作为强制解锁主流程的唯一条件。

### FR-13 运营后台 V1.1

**目标**：运营可维护活动和基础数据。

**需求**：

* 配置红包链接。
* 管理热门菜品库、食材库、品牌转盘。
* 查看数据看板：DAU、推荐点击率、红包点击率、CPS 转化。

## 7. 非功能需求

### 7.1 性能

* 首页首屏加载目标：4G 网络下 2 秒内。
* 推荐接口响应目标：95% 请求 500ms 内。
* MVP 后端并发目标：1000 QPS。

### 7.2 可用性

* 系统可用性目标：99.9%。
* 第三方接口失败时，必须提供缓存或本地菜品降级。
* 用户拒绝定位时，核心随机玩法仍可使用。

### 7.3 安全与隐私

* 所有接口使用 HTTPS。
* openId 不暴露给前端页面。
* 位置权限遵循最小化原则，用户未授权不采集位置。
* 同一用户推荐接口调用需限流，默认单日不超过 100 次。

### 7.4 兼容性

* 支持微信最新版及前两个大版本。
* 适配主流手机分辨率。
* 页面需兼容深色/浅色视觉差异，优先保证可读性。

### 7.5 合规

* 遵守《个人信息保护法》。
* 不做虚假红包宣传。
* 不使用诱导分享文案或强制分享解锁。

## 8. 技术方案初稿

### 8.1 前端

* 微信小程序原生框架。
* 页面结构采用当前项目模板：
  * `pages/index`
  * `pages/logs`
  * 后续新增 `pages/play`
  * 后续新增 `pages/cards`
  * 后续新增 `pages/ingredients`
  * 后续新增 `pages/profile`
  * 后续新增 `pages/menu-list`

### 8.2 后端

候选方案：

* Python FastAPI
* Go

MVP 推荐使用 FastAPI，便于快速迭代推荐规则、地图接口、红包配置和埋点日志。

### 8.3 数据库

* MySQL：用户、偏好、菜单、菜品、反馈、跳转记录。
* Redis：商家 POI 缓存、推荐缓存、限流计数。

### 8.4 第三方接口

* 地图 API：高德或百度地图。
* 外卖 CPS：美团联盟、饿了么分销。
* 微信能力：登录、定位、分享、订阅消息、小程序跳转。

### 8.5 推荐算法

MVP 规则：

```text
score = 0.4 * 评分
      + 0.3 * (1 - 距离 / 最大距离)
      + 0.2 * 人气指数
      + 0.1 * 价格匹配度
```

二期推荐：

* 用户-商家交互矩阵。
* ItemCF 协同过滤。
* GBDT 排序模型。
* 结合用户点赞/点踩反馈持续优化。

## 12. UI/UX 方向

### 12.1 视觉风格

* 主色：`#ff7b42` 活力橙。
* 辅色：暖白、浅棕、深黑背景用于游戏玩法页。
* 组件：圆角卡片、胶囊按钮、横向轮播、柔和阴影。
* 插画：参考图中已有卡通熊/餐盘风格；最终品牌可替换为 Q 版小龙人。

### 12.2 交互原则

* 首页优先减少选择成本，推荐按钮必须明显。
* 玩法页允许更强视觉表现，但结果展示必须清晰。
* 红包入口不应打断推荐主链路。
* 未授权位置时不能阻塞其他玩法。

## 13. MVP 范围

### 13.1 MVP 包含

* 首页基础推荐。
* 附近餐饮入口与模拟/真实 POI 列表。
* 红包卡片轮播与跳转配置。
* 吃喝玩乐活动卡片。
* 转盘玩法。
* 翻卡片玩法。
* 菜单列表与自定义菜单基础管理。
* 食材选择与本地菜品匹配。
* 基础用户登录与偏好设置。
* 基础埋点。

### 13.2 MVP 暂不包含

* 完整机器学习推荐模型。
* 运营后台复杂权限体系。
* CPS 成交回传自动对账。
* 朋友圈海报高级编辑。
* 商家付费推广。

## 15. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|---|---|---|---|
| 地图 API 调用超限 | 中 | 附近餐饮不可用 | Redis 缓存、多服务商备用、默认推荐降级 |
| CPS 政策变化 | 中 | 商业化受影响 | 保留多平台配置，后台可快速上下线 |
| 微信审核风险 | 中 | 上线延期 | 避免诱导分享和虚假红包，提前检查文案 |
| 推荐效果弱 | 中 | 用户留存下降 | 规则兜底、快速收集反馈、AB 测试 |
| 页面玩法过多 | 中 | 首版开发延期 | MVP 聚焦首页、转盘、翻卡片、自定义菜单基础能力 |

## 17. 初稿结论

`AI今天吃什么` 的第一版应以“首页快速推荐 + 红包导流 + 转盘/翻卡片趣味决策 + 自定义菜单 + 食材选择”为核心闭环。参考图片中的页面结构已经形成较清晰的四 Tab 信息架构，建议 MVP 优先复刻该结构，并将 AI 推荐能力先以规则引擎实现，后续再通过反馈数据升级为模型排序。
