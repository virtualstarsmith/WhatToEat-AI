# 优化GLM API调用性能和稳定性

## Goal

优化 `recommend` 云函数中 GLM API 调用的性能和稳定性，解决超时和 socket 连接重置问题，提升用户体验。**关键要求：兜底功能的推荐效果应与大模型推荐效果无差别。同时优化"换一批"功能，控制服务器和大模型成本。确保任何异常场景都直接触发兜底，不显示错误提示。**

## Confirmed Facts

从代码分析确认的事实：

1. **当前实现**（`cloudfunctions/recommend/index.js`）:
   - 传入 GLM 的候选店铺数量：15 个（第 149 行：`topN(scored, 15)`）
   - 超时设置：20 秒（第 97 行：`timeout: 20000`）
   - 每个店铺包含 6 个字段：`poi_id`, `name`, `type`, `distance`, `rating`, `cost`
   - 已有兜底机制：GLM 失败时返回规则评分的 top 3（第 127-133 行）
   - 兜底机制当前缺陷：**没有推荐理由（reason 为空字符串）**
   - **云函数内部错误处理已经返回兜底结果**
   - 错误日志显示：`GLM request timeout after 20s` + `ECONNRESET`

2. **换一批功能**（`pages/index/index.js:189-205`）:
   - 每次点击将当前推荐的 `poi_id` 加入 `excludeIds`
   - `excludeIds` 无限累积，无清理机制
   - **无点击次数限制，用户可无限次点击**
   - **无冷却时间，连续快速点击会频繁调用 GLM API**
   - **对服务器成本和大模型 API 成本造成压力**

3. **前端错误处理问题**（`pages/index/index.js:179-186`）:
   - **当前在 `.catch()` 中显示错误提示："推荐失败，请重试"**
   - **用户需要手动重试，体验不佳**
   - **实际上云函数已经返回了兜底结果，但前端可能没有正确处理**

4. **原始设计要求**（`.trellis/tasks/06-10-wechat-miniprogram-template/design.md:191`）:
   - 请求超时：GLM 调用应设置 8 秒超时
   - 实际代码实现为 20 秒，需回归设计要求

5. **性能目标**（`docs/AI今天吃什么-PRD初稿.md:412`）:
   - 推荐接口响应目标：95% 请求 500ms 内
   - 当前 20 秒超时远超性能目标

6. **错误影响**:
   - 用户等待时间过长
   - socket 连接重置导致请求失败
   - 触发兜底机制时，AI 推荐功能失效
   - **"换一批"功能可能导致成本失控**
   - **前端显示错误提示，用户需要手动重试**

## Requirements

基于确认的事实和用户要求，需要优化以下方面：

1. **减少候选店铺数量** ✅ 已确认
   - 从当前的 15 个减少到 **5-8 个**
   - 平衡推荐质量和响应速度

2. **优化超时设置** ✅ 已确认
   - 从 20 秒减少到 **8 秒**
   - 回归原始设计要求

3. **增强兜底机制质量** ✅ 已确认
   - **兜底功能的推荐效果应与大模型推荐效果无差别**
   - **使用模板生成推荐理由**
   - 基于店铺属性（评分、距离、类型、场景）动态生成自然语言推荐理由

4. **优化"换一批"功能** ✅ 已确认
   - **每天限制 20 次，超过后触发兜底（不再调用 GLM）**
   - **冷却时间 2 秒**
   - **excludeIds 保留最近 2 轮**
   - 提供友好的用户反馈（达到限制时的提示）

5. **前端错误处理优化** ⭐ 新增关键需求
   - **任何异常场景都不显示错误提示**
   - **GLM 超时、失败、网络错误等都直接触发兜底**
   - **确保用户始终能看到推荐结果，无需手动重试**
   - **移除"推荐失败，请重试"的错误提示**

6. **不添加重试机制** ✅ 已确认
   - 依靠兜底机制保障用户体验
   - 快速失败，快速兜底

7. **性能优化**
   - 减少传给 GLM 的数据量
   - 提升响应速度

## Implementation Details

### 模板推荐理由设计

基于店铺属性的模板生成逻辑：

**模板策略**:
- 高评分 + 近距离: "这家店评分 {rating} 分，距离仅 {distance} 米，非常值得尝试"
- 高评分 + 远距离: "虽然距离 {distance} 米，但评分高达 {rating} 分，值得一去"
- 中等评分 + 近距离: "距离仅 {distance} 米，评分 {rating} 分，性价比不错"
- 场景匹配: "正好符合 {场景} 需求，距离 {distance} 米，评分 {rating} 分"

**参数来源**:
- `{rating}`: poi.rating (如果 null，使用"好评"等描述)
- `{distance}`: poi.distance (米，可以转换为"约X分钟步行")
- `{场景}`: 传入的 scene 参数（早餐、午餐、晚餐、夜宵等）
- `{类型}`: poi.type

### "换一批"优化方案

**已确认的策略**:
1. **点击次数限制**: 每天 20 次，超过后不再调用 GLM，直接使用兜底
2. **冷却时间**: 2 秒
3. **excludeIds 历史**: 保留最近 2 轮

**实现逻辑**:
- 使用本地 Storage 记录：
  - 点击次数和时间戳（用于每日重置）
  - 上次点击时间（用于冷却检查）
  - 最近 2 轮的排除记录
- 检查冷却时间（2 秒）和次数限制（20 次/天）
- 超过 20 次后，前端直接调用兜底逻辑，不请求云函数
- excludeIds 维护固定长度数组，超过 2 轮后移除最早的记录

### 前端错误处理优化

**当前问题**:
- `pages/index/index.js:179-186` 中的 `.catch()` 显示错误提示
- 云函数已经返回兜底结果，但前端可能没有正确处理

**优化方案**:
1. **移除错误提示**: 删除 `.catch()` 中的 `error` 状态设置
2. **确保兜底显示**: 云函数返回兜底结果时，前端正常显示推荐
3. **静默处理**: 任何错误都静默触发兜底，用户无感知

**修改文件**:
- `cloudfunctions/recommend/index.js`: 候选数量、超时时间、兜底机制
- `pages/index/index.js`: 点击次数、冷却时间、excludeIds 管理、错误处理优化
