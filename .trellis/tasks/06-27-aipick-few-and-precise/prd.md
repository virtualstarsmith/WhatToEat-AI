# AI甄选：少而准的理性决策

## 背景

甄选现状审计（见父任务）：`0.5距离+0.5评分` 公式排序长列表 → AI 事后写理由。**本质是弱化版美团，AI 零参与决策。** 与美团竞争数据/排序/个性化全面劣势，无差异化。

本任务把甄选从"排序展示长列表"重构为"**AI 推理出少而准的 3-5 家 + 每家理由**"。

## 核心差异化：少而准 = 反平台

```
美团模式：展示附近所有店（50+）→ 用户自己挑（平台要卖广告，必须多展示）
甄选模式：AI 听场景 → 推理出 3-5 家 → 每家有理由（少=决策负担小=价值）
```

**少而准是美团做不了的差异化**——它必须多展示才能商业化。甄选反其道：少而准，AI 替用户承担筛选。

## 范围

### 1. 复用 AI 权重引擎（不重复造轮子）

直接消费子任务2定义的 `utils/aiWeights.js`：
- 场景切换时 `getSceneWeights(scene)` 拿 AI 权重（与抽签同源）
- 甄选用 AI 权重 + 推理，产出 3-5 家

### 2. 重构推荐流程：从"排序"到"推理"

**现状（公式排序）：**
```
pois → scoreCandidates(0.5/0.5) → 按 score 降序 → 取前 N → AI 写理由
```

**重构后（AI 推理）：**
```
pois → scoreCandidates(AI权重) → topN 候选(如10家) → AI 从中推理出 3-5 家 + 理由
```

关键：AI 在**候选池**上做最终选择（推哪 3-5 家），而非事后给排序结果编理由。AI 事前决策。

### 3. AI 推理的 prompt 设计（design 重点）

输入 AI：
- 候选池（topN 的 10 家左右，含名称/类型/距离/评分/人均）
- 当前场景 + AI 权重的 reason（"午餐偏近偏快"）
- 指令：从中选出 3-5 家最适合的，每家给理由

输出：
```json
{
  "recommendations": [
    { "poi_id": "...", "reason": "..." },
    ... 3-5 个
  ]
}
```

复用现有 `callAiRecommend` + `parseRecommendJson`（已支持 recommendations 数组）。

### 4. 结果展示：少而准（UI 调整）

- 推荐数量从"一长串列表"改为 **3-5 家**（design 确定具体数量）
- 每家卡片强化 AI 推荐理由（理由前置、更醒目）
- "换一批"语义保留，但提示"AI 重新为你挑"（强调 AI 决策）
- 候选不足时降级（少于 3 家就展示全部）

### 5. 场景切换的 AI 触发（与抽签一致）

- 场景切换 → 触发 AI 权重计算（复用 aiWeights）+ AI 重新推理推荐
- 有 loading 反馈（"AI 正在为你甄选..."）

## 验收标准

- [ ] 甄选结果为 3-5 家（不再是长列表），每家有 AI 推理理由
- [ ] AI 真正参与选店（从候选池推理出推谁），非事后编理由
- [ ] 场景切换触发 AI 重新推理，有 loading 反馈
- [ ] 复用 aiWeights.js（与抽签同源，单一事实源）
- [ ] AI 失败兜底：回退"距离+评分 top3"（功能不中断）
- [ ] 候选不足（<3）降级展示
- [ ] "换一批"语义改为"AI 重新挑"，强调 AI 决策
- [ ] 与手气抽签的差异化清晰（甄选少而准/多理由 vs 抽签少而玄/强随机）

## Notes

- **核心难点：** AI 推理的稳定性——大模型可能漏掉高分店、选不合理组合。需 design 设计 prompt 约束 + 业务校验（poi_id 必须在候选池内，复用现有 candidateMap 校验）。
- 数量"3-5"需 design 确定具体值（3 最聚焦/5 更全面）。
- 复杂任务，需 design.md（AI 推理流程、prompt、数量策略、兜底、候选不足降级）+ implement.md。
- 依赖子任务2的 aiWeights.js 完成（执行顺序：② 抽签 → ③ 甄选）。
