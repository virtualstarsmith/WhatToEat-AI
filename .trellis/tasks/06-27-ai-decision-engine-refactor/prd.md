# AI决策引擎重构：从文案工到真决策者

## 背景与核心问题（为什么重构）

审计发现：**两个 Tab 的 AI 都是"事后文案工"，没有真正参与选店决策。**

- 手气抽签：`Math.random + 0.4距离+0.4评分+0.2长尾` 公式选店 → AI 只给选好的店编一句理由
- AI甄选：`0.5距离+0.5评分` 公式排序长列表 → AI 只写理由

**选哪家这个核心决策，AI 一毫米都没参与。** 决策者是公式和随机数，AI 只是装饰。App 叫"AI今天吃什么"，但 AI 名存实亡。这正是 PRD §1 吐槽的"同质化"——和美团"距离+评分排序"本质相同，AI 是套了壳的幻觉。

本次重构让 AI 真正进入决策环节。

## 重构方向（已确认）

### 1. 混合模式：AI 打情境分，随机保骨架

不采用"AI 全权选店"（AI 倾向收敛，手气感弱且慢贵）。采用**混合模式**：
- **随机/公式仍主导选店骨架**（保手气感、保速度、保稳定）
- **AI 给候选店打情境调整分**（注入场景智能：午餐抬升快餐、晚餐抬升正餐、值得专程的远店补偿）

**演进说明：** 初版"AI 调三个数值权重"方案已被 sim 证伪（不同权重下结果几乎不变，因距离质量高度相关）。改为 AI 给每家店打情境分——sim 验证有效（午餐/晚餐 top3 零重叠）。详见子任务2 design。

AI 从"事后编理由"升级为"事前注入情境判断"——AI 决定**每家店在当前场景下的情境适配度**，随机/公式在情境分基础上完成具体挑选。

### 2. 触发时机：场景切换时调 AI（缓存）

AI 打情境分的触发点 = **场景切换**（早餐/午餐/下午茶/晚餐/夜宵），而非每次抽签/每次推荐：
- 进页面/切场景 → AI 给当前候选池打情境分 → **缓存**
- 抽签/推荐动作 → 用缓存的情境分瞬时完成（不阻塞）

理由：情境分随场景+候选池变，不随每次抽签变。场景切换时算一次即可，抽签保持瞬时。AI 成本可控（仅场景切换时调用）。

### 3. 两个 Tab 的差异化（感性 vs 理性）

| | 手气抽签（感性） | AI甄选（理性） |
|---|---|---|
| AI 决策方式 | 算场景权重 → 在 AI 权重下强随机选 **1 家** | 算场景权重 → 在 AI 权重下推理出 **3-5 家** |
| 结果数量 | 少则 1 家（极致） | 少则 3-5 家（少而准，反平台） |
| 核心体验 | 手气感（随机+悬念） | 推理感（每家有理由） |
| AI 理由 | 基于AI权重的真实推理（非事后编） | 基于AI权重的真实推理 |

**共同点：两个 Tab 共用同一套"AI 权重引擎"（场景→权重profile），只是消费方式不同（抽签强随机/甄选弱随机+推理）。**

### 4. AI 情境引擎的契约（核心抽象）

这是本次重构最关键的产物——一个统一的、被两个 Tab 共用的模块 `utils/aiContext.js`：

```
输入：候选POI池（精简字段） + scene（场景） + 语境（可选）
输出：{
  adjustments: { [poi_id]: number },  // 每店情境分 -0.3~+0.3
  reason: string                       // AI 对该场景选店倾向的一句话
}
```

- 输入候选池指纹 + 场景 → 缓存（场景/位置不变则不重算）
- 失败兜底：AI 失败 → 返回 null（调用方回退纯公式）
- 抽签和甄选各自消费 adjustments（叠加到 base score），不重复造轮子

## 验收标准（父任务级）

- [ ] 两个 Tab 的选店决策都有 AI 真实参与（非事后编理由）
- [ ] 抽签：场景切换时调 AI 打情境分，抽签动作用情境分瞬时完成
- [ ] 甄选：用 AI 情境分推理出 3-5 家（不再是长列表排序），每家有基于 AI 情境的理由
- [ ] AI 情境引擎 `utils/aiContext.js` 被两个 Tab 共用（单一事实源）
- [ ] AI 失败有兜底（回退纯公式，功能不中断）
- [ ] 筛选栏已移除（子任务1完成 ✅）
- [ ] 手气感保留（抽签仍有随机性，AI 不会让结果过度收敛）
- [ ] 性能：场景切换的 AI 调用有 loading 反馈，抽签/推荐动作本身不阻塞

## Notes

- 这是 App 级根本性重构，优先级 P0。
- 核心难点不在代码，而在**"AI 调权重"如何设计 prompt 让 AI 输出合理 weightsProfile** + **如何让 AI 权重既智能又不破坏手气随机性**。需 design 深入。
- 成本意识：场景切换时调 AI（而非每次抽签），控制调用频次。
- 本父任务本身不做实现，实现工作在各子任务。
